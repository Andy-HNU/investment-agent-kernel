# User Product Feedback 2026-03-30

来源：一次“新用户从建档开始”的手动体验会话，用户未使用仓库内置 demo 人设，而是提供了自己的真实目标与约束。

## 背景

本次体验中，系统侧使用了真实用户输入：

- 账户名：`Andy`
- 当前总资产：`5万`
- 每月投入：`12000`
- 目标金额：`100万`
- 目标期限：`5年`
- 风险偏好：`中等`
- 当前持仓：`现金`
- 最大可接受回撤：`10%`
- 无额外限制

当前仓库可运行的是底层引擎与 demo/report CLI，不是完整的用户前台。因此本次体验中的“提问层”由 Codex 手动扮演。

## 用户反馈与当前状态

### 1. 03 / 05 所需信息是否对用户完全展示

用户问题：

- “用户画像这里，03/05 模块需要的所有信息都展示全了吗？”
- “后续的提问是在推测画像维度数据吗？”

当前状态：

- 没有展示全。
- 当前体验只显式收集了少量用户可直接回答的建档字段。
- `03_snapshot_ingestion` / `05_calibration` 实际消费的输入远多于用户画像本身，还包括：
  - 市场域
  - 账户域
  - 目标域
  - 约束域
  - 行为域
- 本次会话中，除用户明确提供的数据外，剩余字段部分来自系统默认值，部分来自 Codex 代为推断。

结论：

- 当前体验层没有把“用户提供 / 系统推断 / 外部抓取 / 默认填充”的边界讲清楚。
- 这会让用户误以为系统真的“看到了完整输入世界”，但实际上本地仓库目前仍在使用大量默认/模拟数据。

建议动作：

- 增加一层正式 onboarding schema，明确区分：
  - `user_provided`
  - `system_inferred`
  - `externally_fetched`
  - `default_assumed`
- 在建档界面上显式展示“本轮决策到底基于哪些真实输入”。

优先级：

- `P0` 用户透明度问题

### 2. 首份结果泄漏内部策略名，用户不可读

用户问题：

- “当前方案展示了内部的一个策略，用户看不懂的，展开解释一下。”

当前状态：

- 当前基线方案直接展示了类似 `liquidity_buffered__moderate__04` 的内部 candidate 名称。
- 这更像工程侧候选 ID，不是用户可读的方案标题。

结论：

- 当前 `09_decision_card` 还没有把候选方案翻译成真正的用户语言。
- 这会削弱信任感，因为用户看到的是“内部代号”，不是“这套方案为什么适合我”。

建议动作：

- 给每个候选方案增加：
  - 用户可读标题
  - 一句话解释
  - 风险/收益/流动性/复杂度标签
  - 为什么推荐它，不推荐别的
- 把内部 ID 留在 trace/audit 层，而不是直接暴露给用户。

优先级：

- `P0` 决策卡可读性问题

### 3. 应展示多个候选方案与验证证据

用户问题：

- “建议配置里，多给用户一些候选项。”
- “给用户的是不同维度考量的候选项，并且给出每个选项的回测数据表现、验证数据表现。”

当前状态：

- 08/02 已经内部生成候选与排序。
- 但当前面向用户的体验只展示了最终推荐项，没有展开 top-N 候选。
- 当前仓库主要提供的是模型求解结果与风控指标，不是真正的历史回测模块。

结论：

- “候选多样性”在引擎层存在，但没有被展示层消费出来。
- “回测/验证表现”目前只能部分由：
  - success probability
  - shortfall probability
  - max_drawdown_90pct
  - feasibility / confidence notes
  来近似表达。
- 真正的历史回测与样本外验证，还不是当前 repo 的正式能力。

建议动作：

- 在 onboarding / quarterly 卡中展示 top-3 候选：
  - 稳健型
  - 平衡型
  - 进取型
- 每个候选至少展示：
  - 推荐权重
  - success probability
  - shortfall probability
  - max_drawdown_90pct
  - 复杂度标签
  - 约束命中情况
- 如果暂时没有真实 backtest，必须明确写：
  - “以下为模型模拟结果，不是历史回测收益承诺”

优先级：

- `P0` 候选展示
- `P1` 真实回测/验证框架

### 4. 除用户画像外，其余维度输入是模拟的还是实时抓取的

用户问题：

- “这个体验里，除了画像数据以外的几个维度的输入，是模拟的还是实时抓取的？”

当前状态：

- 当前本地仓库里，这些额外维度不是实时抓取。
- 市场、行为、约束等多数仍来自：
  - 系统默认值
  - demo-style builder
  - 本轮会话中的人工填补
- `03_snapshot_ingestion` 具备统一输入与快照化能力，但没有接上真实外部行情/账户/行为数据源。

结论：

- 当前体验必须明确标注“这不是实时投顾，而是本地策略引擎体验”。

建议动作：

- 在每轮决策结果上增加“输入来源标签”：
  - 用户填写
  - 默认假设
  - 手工补录
  - 实时外部源
- 如果没有实时源，不要让用户误解为实时抓取。

优先级：

- `P0` 输入来源透明度

### 5. 目标不可行时，应提供下位替代方案

用户问题：

- “如果发现用户的目标不可行，可以给出下位替代吗？我觉得可以是一个 v2 的特性。”

当前状态：

- 当前系统在不可行时会给：
  - `no_feasible_allocation`
  - `closest_feasible_candidate`
  - `reassess_goal_amount_or_horizon_or_drawdown_or_candidate_allocations`
- 也就是说，系统已经能发现“目标过紧”，但还不会自动产出结构化替代方案梯子。

结论：

- 这是非常合理的 `V2` 方向。
- 当前提示过于工程化，用户知道“不太行”，但不知道“下一步怎么改最有效”。

建议动作：

- 增加 `goal fallback ladder` / `goal alternatives`：
  - 方案 A：维持目标金额，期限从 `5年 -> 6年/7年`
  - 方案 B：维持期限，目标从 `100万 -> 90万/85万`
  - 方案 C：维持目标与期限，月投入从 `12000 -> 15000`
  - 方案 D：维持目标与期限，允许回撤从 `10% -> 15%`
- 每个替代方案都给出新的 success probability 与风险指标。

优先级：

- `P1`，但用户价值很高，建议尽早排入

## 建议汇总

这组反馈本质上不是“算法还不够”，而是“用户前台层还没有把底层引擎能力正确交付出来”。

建议把后续工作拆成两层：

### Layer 1: Frontend / Interaction Truthfulness

- 明确哪些输入是用户填的
- 明确哪些输入是系统推断的
- 明确哪些输入来自默认值
- 明确哪些输入来自实时外部源

### Layer 2: Decision Explainability

- 不再向用户直接暴露内部 candidate ID
- 展示 top-N 候选与对比理由
- 补“不可行目标的替代路径”

## 建议优先级

### 必做

- onboarding schema 与输入来源标注
- 候选方案用户可读化
- top-N 候选展示

### 可作为 V2

- goal alternatives / fallback ladder
- 真正的 backtest / validation evidence 层
