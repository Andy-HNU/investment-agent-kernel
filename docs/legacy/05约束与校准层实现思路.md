# 05_constraint_and_calibration.md

# 约束与校准层设计思路 v1

> **文档定位**：本文件描述系统的约束与校准层（Constraint & Calibration Layer）的作用、边界、输入输出、实现思路与 v1 范围。  
> 它是 `03_snapshot_and_ingestion.md` 的直接下游，是 `02_goal_solver.md`、`04_runtime_optimizer.md` 与 `10_ev_engine.md` 的参数治理层与状态解释层。  
> 它不直接参与 workflow 编排，不直接对用户展示，不直接负责原始外部采集。

---

## 0. 一句话定义

**Constraint & Calibration Layer 是系统的解释与治理中枢。**

它负责基于 03 提供的原始快照：

- 解释市场输入
- 构造系统内部状态
- 校准长期求解参数
- 校准运行期阈值与约束
- 管理参数回写与版本更新

它不直接回答“最终动作选哪个”，但它决定：

> **系统应当以什么样的市场理解、什么样的约束口径、什么样的参数版本来做后续判断。**

---

## 1. 为什么需要单独做 05

03 只能把输入世界统一化，但不能回答“这些输入在系统里意味着什么”。

举例来说：

- 03 可以提供价格序列和收益率序列
- 但不能决定这些序列应解释成怎样的 `expected_returns`、`volatility`、`correlation_matrix`
- 03 可以提供约束原文和规则配置
- 但不能决定当前运行期应该使用哪个 drawdown 阈值、哪个行为惩罚系数
- 03 可以提供行为日志
- 但不能决定当前系统是否应提高 `BehaviorPenalty`

因此 05 的意义是：

> **把“原始输入”解释成“系统正式输入”，并持续维护这些正式输入的参数口径。**

如果没有 05，会出现三个问题：

### 1.1 参数逻辑四散
Goal Solver、Runtime、EV 都会偷偷各自长出“自己的市场理解”和“自己的阈值逻辑”。

### 1.2 长期与短期口径脱节
长期求解和运行期优化会基于两套不一致的市场理解工作。

### 1.3 约束回写无处安放
系统运行后的复盘、阈值调整、偏差回灌，没有稳定归属。

---

## 2. 05 在系统中的作用

05 的作用建议概括为三层。

### 2.1 状态解释
把 03 的原始快照解释成系统内部正式状态，例如：

- `MarketState`
- `ConstraintState`
- `BehaviorState`

### 2.2 参数校准
将原始市场输入与历史运行反馈转化为：

- `MarketAssumptions`
- 校准后的 `GoalSolverParams`
- 校准后的 `EVParams`
- 校准后的 `RuntimeOptimizerParams`
- 校准后的风险阈值与预算

### 2.3 参数治理
管理参数的来源、版本、更新时间、适用范围与回写策略，保证系统运行时使用的是一套一致、可追踪的解释口径。

---

## 3. 05 的系统位置

推荐链路如下：

`03_snapshot_and_ingestion`  
→ `05_constraint_and_calibration`  
→ `07_orchestrator_workflows`  
→ `02_goal_solver / 04_runtime_optimizer / 10_ev_engine`

其中：

- 03 负责原始输入定版
- 05 负责解释与校准
- 07 负责 workflow 路由与注入
- 02/04/10 负责消费

## 4. 05 的职责边界

### 4.1 05 负责的内容

05 负责：

- 消费 03 的原始快照与 bundle
- 从原始市场输入构造 `MarketState`
- 从原始市场输入与校准逻辑构造 `MarketAssumptions`
- 从约束原文与系统反馈构造 / 更新 `ConstraintState`
- 从行为日志与系统反馈构造 / 更新 `BehaviorState`
- 更新 Goal Solver / Runtime / EV 所使用的参数版本
- 管理参数回写、版本、适用周期、更新时间和质量标记

### 4.2 05 不负责的内容

05 不负责：

- 外部市场数据采集
- 账户系统直连
- 原始日志直接抓取
- workflow 触发编排
- 候选动作生成
- Goal Solver 求解执行
- EV 评分执行
- 决策卡展示文案生成

### 4.3 与 03 的关系

03 负责“把世界拿进来并冻结”。  
05 负责“解释这些输入在系统里意味着什么”。

推荐明确写为：

> 03 不直接生成 MarketState / MarketAssumptions；  
> 05 不直接回头访问原始外部源。

这样分层最稳。

---

## 5. 05 的核心问题是什么

05 要解决的不是“拉数据”，而是三个解释问题。

### 5.1 市场解释问题
原始市场输入如何变成系统理解下的市场状态与长期假设。

### 5.2 约束解释问题
原始 IPS、软偏好、预算约束、结构边界，如何变成运行期可执行的约束状态。

### 5.3 行为解释问题
原始行为日志与干预历史，如何变成系统实际使用的行为惩罚与行为状态。

所以 05 的本质不是一个单纯“调参数”的文件，而是：

> **输入解释 + 参数治理 + 回写更新 的中间层。**

---

## 6. 05 的核心输出

05 的正式输出建议分成两类：

### 6.1 状态类输出
这些主要给 Runtime / EV 用：

- `MarketState`
- `ConstraintState`
- `BehaviorState`

### 6.2 参数类输出
这些主要给 Goal Solver / Runtime / EV 用：

- `MarketAssumptions`
- 更新后的 `GoalSolverParams`
- 更新后的 `RuntimeOptimizerParams`
- 更新后的 `EVParams`
- 各类风险阈值 / 行为惩罚系数 / 预算限制

---

## 7. 05 的子问题拆分

05 建议显式拆成四个子职责，不然 Claude 很容易写成一团。

### 7.1 Market Interpretation
输入：03 的市场域原始快照  
输出：`MarketState`

这里解决的是：

- 当前市场输入应如何被系统“看见”
- 哪些字段属于运行期市场状态
- 这些状态是否足够完整、是否需要降级

注意：

- `MarketState` 更偏运行期状态
- 它不等同于 `MarketAssumptions`
- 它也不直接等于外部原始行情

### 7.2 Assumption Calibration
输入：市场快照、历史反馈、既有参数版本  
输出：`MarketAssumptions`

这里解决的是：

- Goal Solver 长期求解所需的期望收益、波动、相关性口径从哪里来
- 这些参数多久更新
- 采用怎样的更新节奏和保守机制
- 是否允许部分沿用旧版本

这一层应明确与 02 对接：

> Goal Solver 不修改 `market_assumptions`，只消费它。

### 7.3 Constraint Calibration
输入：原始约束快照、历史执行反馈、回撤/偏离等运行结果  
输出：`ConstraintState` 及相关阈值更新

这里解决的是：

- 原始 IPS 文本 / 规则配置如何变成运行期约束状态
- 某些阈值是否需要被调紧或调松
- 再平衡带宽、卫星上限、drawdown 阈值等是否应更新

### 7.4 Behavior Calibration
输入：行为快照、历史 override、运行反馈  
输出：`BehaviorState` 与行为惩罚相关参数

这里解决的是：

- 用户近期行为在系统里应如何表示
- 是否应触发更强的行为惩罚
- 是否应提高冷静期、降低动作激进度、限制部分建议

## 8. MarketState 与 MarketAssumptions 的区别

这一节必须写清楚，这是 03/05/02/04/10 对接的核心。

### 8.1 MarketState 是“当前运行期视角”
它描述的是：

- 当前市场输入在运行期下的解释结果
- 更偏短周期、当前轮次、可用于 Runtime / EV 的状态

例如它可以承载：

- 当前风险环境标签
- 流动性状态
- 估值位置
- 波动状态
- 当前相关性风险提示
- 市场状态质量标记

### 8.2 MarketAssumptions 是“长期求解视角”
它描述的是：

- Goal Solver 在 Monte Carlo 中所采用的长期求解参数口径

例如它可以承载：

- expected returns
- volatility
- correlation matrix
- 更新版本
- 生效周期
- 估计质量标记

### 8.3 两者的关系
两者同源，但不相同。

- 它们都来自 03 的市场域快照
- 都由 05 构造
- 但服务对象不同

建议明确一句：

> `MarketState` 服务运行期；  
> `MarketAssumptions` 服务长期求解；  
> 二者不得混用，也不应由下游模块自行互相推断。

## 9. 05 的实现思路

### 9.1 整体方向

05 的 v1 不应做成一个“聪明但不可控的大黑盒”，而应做成一个：

- 口径明确
- 版本可追踪
- 更新节奏受控
- 降级可解释

的解释与校准层。

05 的首要目标不是“模型最先进”，而是：

> **先让系统的正式输入口径统一、稳定、可回写。**

### 9.2 三段式实现逻辑

#### 第一段：解释
从 03 提供的 bundle 中抽取各域原始输入，生成：

- `MarketState`
- `ConstraintState`
- `BehaviorState`

#### 第二段：校准
根据预定义规则、历史反馈、已有参数版本，更新：

- `MarketAssumptions`
- `GoalSolverParams`
- `RuntimeOptimizerParams`
- `EVParams`

#### 第三段：发布
将上述状态和参数形成一个统一的“校准结果包”，供 07 注入下游模块。

## 10. 05 的参数回写逻辑

05 的一个关键职责是“回写”，但这里的回写不是随意修改，而应是受控治理。

### 10.1 回写对象
包括但不限于：

- 期望收益 / 波动 / 相关性
- drawdown 阈值
- satellite 上限
- 行为惩罚系数
- 再平衡带宽
- 预算约束参数

### 10.2 回写来源
可来自：

- 原始输入的解释结果
- 历史运行反馈
- 复盘结果
- 参数偏离监测
- 人工确认的策略更新

### 10.3 回写原则
建议明确几条硬原则：

- 不允许无版本回写
- 不允许静默覆盖旧参数
- 不允许下游模块自行写回
- 不允许把一次临时降级结果永久写死为默认参数
- 参数更新必须带来源、时间、适用范围与质量说明

---

## 11. 05 的降级与保守策略

05 必须支持不完整输入下的降级，但降级必须可解释。

### 11.1 可降级的情况
例如：

- 行为域缺失
- 某些市场标签缺失
- 某些辅助约束字段缺失
- 某次更新窗口不足

### 11.2 不宜强行降级的情况
例如：

- 核心约束字段缺失
- 长期求解所需关键市场假设无法构造
- 目标求解依赖的核心参数版本不存在
- 数据质量明显失真

### 11.3 降级原则
建议明确：

- 降级优先于静默伪造
- 沿用旧版本优先于临时拍脑袋重建
- 强约束优先于激进优化
- 运行期建议应服从参数质量边界

---

## 12. 05 的版本治理思路

05 不是一次性计算器，而是参数治理层，所以必须显式保留版本概念。

### 12.1 需要版本化的对象
- `MarketAssumptions`
- `GoalSolverParams`
- `RuntimeOptimizerParams`
- `EVParams`
- 关键阈值与预算

### 12.2 版本信息至少应包含
- version id
- source bundle id
- 生效时间
- 更新时间
- 更新原因
- 质量标记
- 是否为默认 / 临时 / 人工确认版本

### 12.3 版本治理目标
让系统未来能回答：

- 当前参数是从哪一轮输入推导来的
- 它为什么被更新
- 它何时生效
- 它是否在降级模式下生成
- 旧版本是否还能回放复盘

## 13. 05 与其他模块的关系

### 与 `03_snapshot_and_ingestion.md`
03 提供原始快照；05 是其第一解释层。  
05 不直接抓原始数据。

### 与 `02_goal_solver.md`
05 负责生成并维护 `MarketAssumptions` 与相关求解参数；02 只消费，不自行更改。

### 与 `04_runtime_optimizer.md`
05 提供 Runtime 所需的 `MarketState`、`ConstraintState`、`BehaviorState` 与相关参数版本。  
04 不应内嵌参数校准公式。

### 与 `07_orchestrator_workflows.md`
07 负责在某轮 workflow 中选择合适的 bundle 和 calibration result，并注入下游模块。  
07 不负责定义参数公式。

### 与 `10_ev_engine.md`
05 提供 EV 所需的参数与约束口径，但不参与 EV 实际打分。  
EV 不应自行定义另一套市场解释或阈值体系。

## 14. 05 v1 范围

### 14.1 v1 应做到的
- 明确 05 是 03 的下游解释层
- 明确 `MarketState` 与 `MarketAssumptions` 的区别
- 明确 `ConstraintState` 与 `BehaviorState` 的生成归属
- 明确参数回写与版本治理职责
- 明确与 02 / 04 / 10 / 07 的接口边界
- 支持最小降级与保守回退策略

### 14.2 v1 不应急着做的
- 全自动复杂自学习校准
- 黑盒机器学习参数调节器
- 高度自适应的动态 regime 模型
- 将 05 做成“万能策略中枢”
- 把工作流触发逻辑混进 05
- 把用户展示逻辑混进 05

---

## 15. 05 的验收标准

05 的验收重点不应是“看起来聪不聪明”，而应是：

### 15.1 口径统一性
长期求解与运行期优化是否基于同一套解释体系。

### 15.2 边界清晰性
05 是否没有越界去做采集、编排、求解、评分。

### 15.3 参数可追踪性
参数是否有清晰版本、来源、更新时间、质量标记。

### 15.4 降级可解释性
输入不完整时，05 是否给出受控、保守、透明的降级结果。

### 15.5 回写受控性
参数更新是否可审计、可回放、可理解。

---

## 16. 一句话收束

> **05 的价值，不是“调几个参数”，而是把原始输入解释成系统真正使用的正式状态与正式参数，并让这些口径长期保持统一、可追踪、可回写。**
