# 09_decision_card_spec.md

# 决策卡层实现思路 v1

> **文档定位**：本文件描述系统的决策卡层（Decision Card Layer）的作用、边界、输入输出、实现思路与 v1 范围。  
> 它消费 `02_goal_solver.md`、`04_runtime_optimizer.md`、`10_ev_engine.md` 与 `07_orchestrator_workflows.md` 输出的结构化结果，把系统已经完成的判断翻译成用户可读、可审计、可执行的展示对象。  
> 它不直接做目标求解，不直接做 EV 打分，不直接决定 workflow，只负责**把已经得到的结论与证据正确地展示出来**。

---

## 0. 一句话定义

**Decision Card 是系统的结果展示层。**

它不负责回答“该不该买”，也不负责回答“哪个动作得分更高”。
它只做一件事：

> **把 Goal Solver、Runtime Optimizer、EV Engine 与 Orchestrator 已经形成的结构化结论，翻译成一张用户能读懂、能质疑、能执行、能复盘的决策卡。**

---

## 1. 为什么需要单独做 09

如果没有 09，系统会出现三类问题。

### 1.1 解释散落到各业务层

04 会想顺手拼一句推荐文案。  
10 会想顺手多写一点“为什么”。  
02 会想顺手把配置结论直接展示出来。

结果就是：

- 文案口径散
- 展示字段不统一
- 同样的建议在不同链路里说法不一致

### 1.2 用户只能看到结论，看不到证据

系统真正的价值，不是“神秘地给出一个动作”，而是要让用户看到：

- 推荐什么
- 不推荐什么
- 为什么是这个而不是那个
- 这条建议依赖哪些假设与边界

如果没有单独的 09，这部分会被忽略，系统就会显得像在瞎说。

### 1.3 展示层会反向污染策略层

如果让 04/10 直接承担 UI/文案，就很容易为了“好展示”而修改策略字段和评分结果。  
09 独立出来之后，策略层只管生成结构化事实，展示层只管表达。

---

## 2. 09 在系统中的作用

09 的作用建议概括为三层。

### 2.1 结果翻译

把上游结构化对象翻译成用户可读内容。

### 2.2 证据编排

把推荐理由、关键数字、边界条件、阻断原因、备选动作，组织成固定结构，而不是随意堆文字。

### 2.3 复盘锚定

让每一张卡都能回指到：

- 哪一轮 workflow
- 哪一版输入与参数
- 哪一个推荐动作或阻断结论

---

## 3. 09 的系统位置

推荐链路如下：

`07_orchestrator_workflows`  
→ `09_decision_card_spec`

09 的直接上游并不是用户输入，也不是外部市场数据，而是：

- `GoalSolverOutput`
- `RuntimeOptimizerResult`
- `EVReport`
- `OrchestratorOutput` 提供的 card build input

因此，09 更准确的定位是：

> **系统正式判断的最后一层解释与展示壳。**

---

## 4. 09 的职责边界

### 4.1 09 负责的内容

09 负责：

- 消费结构化上游结果
- 决定本轮卡片采用哪种展示模板（由 07 明确传入 card type）
- 生成标题、状态标签、主建议、证据区、边界区、备选动作区、追踪信息区
- 组织推荐理由与不推荐理由
- 明确展示低置信度、阻断、降级、冷静期等特殊状态
- 给出结构化卡片对象，供前端或消息层展示

### 4.2 09 不负责的内容

09 不负责：

- workflow 路由
- 候选动作生成
- EV 打分
- Goal Solver 求解
- 参数校准
- 事件识别
- 持久化策略日志
- 任何数值的二次计算与二次改写

### 4.3 与 07 的关系

推荐明确写为：

> 07 决定“这轮该生成什么类型的卡”；  
> 09 决定“这张卡长什么样、怎么说清楚”。

这样边界最稳。

---

## 5. 09 的核心问题是什么

09 要解决的，不是“做一个 UI 组件”，而是四个解释问题。

### 5.1 这一轮到底要告诉用户什么

- 是给基线配置解释
- 还是给运行期动作建议
- 还是告诉用户“本轮被阻断，不能给建议”
- 还是告诉用户“建议先观察，不建议动作”

### 5.2 哪些证据必须露出来

用户至少需要看到：

- 主建议
- 推荐原因
- 关键数字
- 不推荐其他方案的原因
- 风险边界与执行限制

### 5.3 哪些东西不能乱说

09 不能补造：

- 上游没有给出的概率数字
- 上游没有给出的风险判断
- 上游没有给出的参数变化原因
- 上游没有给出的执行条件

### 5.4 如何把“别动”也说清楚

系统里 `FREEZE`、`OBSERVE`、blocked、degraded 这些状态很重要。  
09 必须把“为什么现在不建议动作”解释清楚，而不是只在用户看到动作时才有卡片价值。

## 6. 09 的核心输入

09 的输入建议统一为一个 card build input，不要让前端或调用方直接拼多种对象。

### 6.1 建议的输入对象

```python
from dataclasses import dataclass
from typing import Literal

class DecisionCardType(Enum):
    GOAL_BASELINE = "goal_baseline"
    RUNTIME_ACTION = "runtime_action"
    QUARTERLY_REVIEW = "quarterly_review"
    BLOCKED = "blocked"

@dataclass
class DecisionCardBuildInput:
    card_type: DecisionCardType
    workflow_type: str                     # onboarding / monthly / event / quarterly
    run_id: str
    bundle_id: str | None
    calibration_id: str | None

    goal_solver_output: GoalSolverOutput | None = None
    runtime_result: RuntimeOptimizerResult | None = None

    blocking_reasons: list[str] = field(default_factory=list)
    degraded_notes: list[str] = field(default_factory=list)
    escalation_reasons: list[str] = field(default_factory=list)
```

### 6.2 为什么要统一输入对象

因为 09 真正关心的是“展示场景”，不是“上游函数签名长什么样”。  
把输入统一成一个 build input 后：

- 07 负责把场景准备好
- 09 负责按场景取数并渲染

这样最不容易污染边界。

---

## 7. 09 的核心输出

09 的正式输出不应是一坨字符串，而应是一份结构化卡片对象。

### 7.1 建议的输出对象

```python
@dataclass
class DecisionCard:
    card_id: str
    card_type: DecisionCardType
    workflow_type: str

    title: str
    status_badge: str                  # ok / caution / blocked / observe / degraded
    summary: str                       # 一句话摘要

    primary_recommendation: str        # 主建议
    recommendation_reason: list[str]   # 推荐理由（多条）
    not_recommended_reason: list[str]  # 不推荐其他动作/配置的理由

    key_metrics: dict[str, str]        # 展示用关键指标，已格式化
    alternatives: list[str]            # 备选动作/配置摘要
    guardrails: list[str]              # 不应做什么 / 本轮限制
    execution_notes: list[str]         # 执行条件、冷静期、资金来源等

    trace_refs: dict[str, str]         # run_id / bundle_id / calibration_id / snapshot_id 等
```

### 7.2 输出对象的作用

它既可以给前端直接渲染，也可以给消息层、报告层或历史复盘页复用。  
09 不应只输出一段不可结构化的长文案。

---

## 8. 09 的卡片类型拆分

v1 建议至少支持四类卡片。

### 8.1 基线配置解释卡（GOAL_BASELINE）

适用场景：

- 首次建档
- 需要重新建立长期基线时

上游依赖：

- `GoalSolverOutput`
- 可选的候选配置摘要（若 07 传入）

这张卡至少要回答：

- 推荐配置是什么
- 成功概率大概处于什么水平
- 为什么不是选更激进或更保守的配置
- 当前主要风险在哪

建议展示字段：

- `recommended_allocation.name`
- `recommended_result.success_probability`
- `risk_summary.max_drawdown_90pct`
- `risk_summary.shortfall_probability`
- `structure_budget` 摘要
- `solver_notes`

核心要求：

> **不能只告诉用户“建议 55/30/15”，而要告诉用户“为什么是这套”。**

### 8.2 运行期动作卡（RUNTIME_ACTION）

适用场景：

- 月度巡检
- 事件触发
- 季度复审后的动作建议

上游依赖：

- `RuntimeOptimizerResult`
- 其中的 `EVReport`

这张卡至少要回答：

- 这一轮推荐动作是什么
- 推荐置信度如何
- 为什么选这个动作
- 被淘汰动作为什么不行
- 本轮有哪些执行限制

建议展示字段：

- `ev_report.recommended_action`
- `ev_report.recommended_score`
- `ev_report.confidence_flag / confidence_reason`
- `ev_report.goal_solver_baseline`
- `ev_report.goal_solver_after_recommended`
- top1 / top2 摘要
- `eliminated_actions` 中的主要 fail reasons
- `candidate_poverty`

### 8.3 季度复审卡（QUARTERLY_REVIEW）

适用场景：

- QUARTERLY 模式完成新基线重算后

上游依赖：

- `GoalSolverOutput`
- `RuntimeOptimizerResult`

这张卡不是简单地把基线卡和动作卡拼一起，而应该突出：

- 新基线相对旧基线是否发生变化
- 本季度动作建议是否与新基线一致
- 是否存在参数或预算收紧/放松

建议展示字段：

- 新基线成功概率
- 结构预算变化摘要
- 推荐动作与季度状态标签
- `trigger_type = quarterly`

### 8.4 阻断 / 降级卡（BLOCKED）

适用场景：

- `bundle_quality == DEGRADED`
- `calibration_quality == degraded`
- 强约束冲突导致本轮无法给建议
- 07 判定必须阻断或升级人工复审

上游依赖：

- `OrchestratorOutput.blocking_reasons`
- `degraded_notes`
- 必要的追踪信息

这张卡至少要回答：

- 本轮为什么没有形成正式建议
- 问题来自哪一层
- 现在允许用户做什么、不允许做什么
- 下一步应该补什么信息或等待什么条件

## 9. 09 的构建思路

### 9.1 整体方向

09 的 v1 不应实现成一个“花里胡哨的前端模板库”，而应实现成一个：

- 场景明确
- 字段稳定
- 解释清楚
- 不造数据
- 易于复盘

的结果解释层。

09 的首要目标不是“说得多漂亮”，而是：

> **让用户能看懂系统到底在建议什么、为什么建议、为什么不建议别的。**

### 9.2 三段式实现逻辑

#### 第一段：判模板
根据 `card_type` 决定用哪种卡片模板。

#### 第二段：抽事实
从 `GoalSolverOutput / RuntimeOptimizerResult / EVReport / blocking_reasons` 中抽取正式事实。

#### 第三段：组装表达
把事实放进固定区域：标题、主建议、证据、限制、备选、追踪信息。

---

## 10. 09 的建议接口

```python
def build_decision_card(inp: DecisionCardBuildInput) -> DecisionCard:
    """
    09 的唯一正式入口。

    输入：
    - 由 07 准备好的 card build input

    输出：
    - 结构化决策卡对象
    """
    ...
```

### 10.1 伪代码主流程

```python
def build_decision_card(inp: DecisionCardBuildInput) -> DecisionCard:
    if inp.card_type == DecisionCardType.GOAL_BASELINE:
        return _build_goal_baseline_card(inp)

    if inp.card_type == DecisionCardType.RUNTIME_ACTION:
        return _build_runtime_action_card(inp)

    if inp.card_type == DecisionCardType.QUARTERLY_REVIEW:
        return _build_quarterly_review_card(inp)

    if inp.card_type == DecisionCardType.BLOCKED:
        return _build_blocked_card(inp)

    raise ValueError(f"unsupported card_type: {inp.card_type}")
```

## 11. 09 的字段映射思路

这一节必须写清楚，否则 09 很容易自己瞎补口径。

### 11.1 基线配置解释卡的取数来源

- 标题：来自 workflow_type + `GoalCard.goal_description`
- 主建议：来自 `GoalSolverOutput.recommended_allocation`
- 核心指标：来自 `recommended_result.success_probability / risk_summary`
- 结构预算：来自 `structure_budget`
- 说明：来自 `solver_notes`

### 11.2 运行期动作卡的取数来源

- 标题：来自 `RuntimeOptimizerResult.mode`
- 主建议：来自 `ev_report.recommended_action`
- 推荐理由：来自 `recommended_score` + top1 的 `recommendation_reason`
- 不推荐理由：来自 top2 差异与 `eliminated_actions`
- 关键指标：来自 `goal_solver_baseline / goal_solver_after_recommended / confidence_flag`
- 限制与注意事项：来自 `candidate_poverty / confidence_reason / action.cooldown_applicable`

### 11.3 阻断卡的取数来源

- 标题：来自 `workflow_type`
- 状态标签：来自 blocked / degraded
- 原因：来自 `blocking_reasons / degraded_notes`
- 下一步动作：由 07 传入的控制建议（如“补数据”“等待人工确认”）

## 12. 09 的文案规则

09 必须约束自己，不允许变成“自由发挥器”。

### 12.1 必须遵守的规则

- 只能使用上游已经给出的事实与字段
- 数值缺失时可以省略，不可以补造
- 置信度低时必须显式说低，不可包装成确定建议
- `FREEZE / OBSERVE` 也必须给出原因
- blocked 卡不得偷偷附带动作建议
- 推荐动作与不推荐动作都应尽量给出一句解释

### 12.2 推荐的表达风格

推荐固定分成五段：

1. **这轮是什么**
2. **主建议是什么**
3. **为什么是它**
4. **为什么不是别的**
5. **本轮边界与限制是什么**

### 12.3 不应出现的表达

- “相信系统”
- “大概率一定会”
- “市场就是会这样走”
- 任何上游没有给出的额外承诺

## 13. 09 的特殊状态处理

### 13.1 推荐动作为 `FREEZE / OBSERVE`

这不是失败，也不是空白卡。  
09 应明确告诉用户：

- 当前改善空间有限
- 或当前动作受行为/约束限制
- 因此维持现状 / 观察是更优动作

### 13.2 低置信度

若 `confidence_flag == low` 或 `candidate_poverty == True`，卡片必须显式带：

- 低置信度标签
- 原因说明
- 建议用户不要把本轮建议理解为强动作信号

### 13.3 阻断 / 降级

当 workflow 已被 07 阻断或降级时，09 不应试图“补一张正常建议卡”。  
应该输出专门的 blocked / degraded card。

## 14. 09 与其他文档的关系

### 与 `02_goal_solver.md`
09 消费 Goal Solver 的结构化输出做基线解释。  
09 不评价 Goal Solver 算法本身。

### 与 `04_runtime_optimizer.md`
09 消费 `RuntimeOptimizerResult`，尤其是其 `ev_report` 与运行元数据。  
09 不修改运行期结果。

### 与 `07_orchestrator_workflows.md`
07 决定场景与卡片类型；09 负责渲染。  
09 不决定 workflow。

### 与 `10_ev_engine.md`
09 消费 `EVReport` 的排序、推荐动作、分项得分与置信度。  
09 不碰评分逻辑。

## 15. 09 v1 范围

### 15.1 v1 应做到的

- 明确四类正式决策卡
- 明确统一的 `DecisionCardBuildInput`
- 明确 `DecisionCard` 结构化输出
- 明确字段映射来源
- 明确低置信度 / blocked / observe 的展示规则
- 明确 09 不能补造数字与判断

### 15.2 v1 不应急着做的

- 高度个性化的自然语言风格迁移
- 复杂图表工厂
- 多端 UI 自适配规范
- 富交互前端协议
- 在 09 做任何二次评分或二次求解

---

## 16. 09 的验收标准

09 的验收重点不应是“文案漂不漂亮”，而应是：

### 16.1 解释完整性
用户是否能看懂：推荐什么、为什么推荐、为什么不推荐别的。

### 16.2 字段一致性
不同 workflow 下，相同概念是否使用相同口径。

### 16.3 边界干净性
09 是否没有越界去做求解、评分、编排与持久化。

### 16.4 特殊状态可解释性
`FREEZE / OBSERVE / low confidence / blocked / degraded` 是否都能被清楚表达。

### 16.5 可追踪性
每张卡是否都能回指到本轮 run、bundle、calibration 与 solver/runtime 结果。

---

> **09 的价值，不是“把结论写成人话”，而是把系统已经形成的正式判断，连同其证据、边界和限制，一起交付给用户。**
