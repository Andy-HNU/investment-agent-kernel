# 07_orchestrator_workflows.md

# 编排与工作流层实现思路 v1

> **文档定位**：本文件描述系统的编排与工作流层（Orchestrator & Workflows）的作用、边界、输入输出、实现思路与 v1 范围。  
> 它位于 `03_snapshot_and_ingestion.md` 与 `05_constraint_and_calibration.md` 之后，负责把原始输入世界、校准结果、长期求解、运行期优化与决策卡展示串成一条可执行闭环。  
> 它不直接做目标求解，不直接做 EV 打分，不直接做决策卡文案，只负责**何时触发、调谁、是否阻断、如何回写**。

---

## 0. 一句话定义

**Orchestrator 是系统的运行总控层。**

它不负责判断“哪个配置更优”，也不负责判断“哪个动作得分更高”。
它只做一件事：

> **在正确的时机，基于正确版本的输入与参数，调用正确的下游模块，控制阻断/降级/升级，并把本轮运行结果绑定成一条可回放的系统执行链。**

---

## 1. 为什么需要单独做 07

如果没有 07，这套系统会出现四类典型问题。

### 1.1 谁来决定什么时候跑哪条链路

03 只负责把输入世界冻结。  
05 只负责解释与校准。  
02 只负责回答“够不够”。  
04/10 只负责回答“怎么改更优”。

但系统真正运行时，总要有人决定：

- 这次是首次建档、月度巡检、事件升级还是季度复审
- 这一轮该不该重跑 Goal Solver
- 这一轮是不是只允许走运行期优化
- 这一轮数据质量是否允许继续

这个职责如果没人接，就会散落到各层，最后每层都偷偷带一点 workflow 逻辑。

### 1.2 谁来做阻断与降级判断

03 会打 `bundle_quality`。  
05 会打 `calibration_quality`。  
04 会打 `candidate_poverty`。  
但它们都不应该自己决定“系统这一轮到底停不停、降不降、升不升级”。

因此必须有一层专门承担：

> **把各模块打出的质量与风险标志，翻译成正式的运行控制决策。**

### 1.3 谁来保证跨模块版本对齐

系统里至少存在以下几种版本锚点：

- `SnapshotBundle.bundle_id`
- `GoalSolverInput.snapshot_id`
- `CalibrationResult.calibration_id`
- `GoalSolverParams.version`
- `RuntimeOptimizerParams.version`
- `EVParams.version`
- 本轮运行自己的 `run_id`

如果没有 07 统一绑定，后续复盘时会回答不了两个关键问题：

- 这一轮建议到底基于哪一份输入世界
- 这一轮动作建议到底用了哪一版参数和哪一版基线

### 1.4 谁来承接持久化与状态回写

04 已明确不负责持久化；05 也不负责 workflow 触发；09 更不负责日志。  
因此系统里必须有一层对“本轮发生了什么”负责落账。

## 2. 07 在系统中的作用

07 的作用建议概括为四层。

### 2.1 触发与路由

识别当前触发源，决定进入哪条 workflow。

### 2.2 串联与注入

把 03、05、08、02、04、09 按正确顺序串起来，把上游结果注入下游模块。

### 2.3 阻断与升级

根据 `bundle_quality`、`calibration_quality`、`candidate_poverty`、冷静期/行为状态等信号，决定：

- 是否阻断本轮
- 是否允许降级继续
- 是否需要升级为人工复审
- 是否要把推荐动作降成 `FREEZE / OBSERVE`

### 2.4 回写与追踪

把本轮运行的关键输入引用、参数版本、求解结果、运行结果、决策卡结果和执行状态，统一写入审计链与复盘链。

## 4. 07 的职责边界

### 4.1 07 负责的内容

07 负责：

- 识别触发源与触发类型
- 选择 workflow（首次建档 / 月度巡检 / 事件触发 / 季度复审）
- 调用 03 构建 `SnapshotBundle`
- 调用 05 生成 `CalibrationResult`
- 根据 `bundle_quality` 与 `calibration_quality` 决定阻断或降级
- 在需要时调用 08 生成 `candidate_allocations`
- 在需要时调用 02 完整重算 Goal Solver
- 在运行期链路中调用 04 生成 `RuntimeOptimizerResult`
- 根据本轮上下文组装给 09 的 `DecisionCardBuildInput`
- 落账本轮运行日志、状态引用、参数版本、卡片结果与执行状态
- 根据 `candidate_poverty`、阻断标记、手动请求等信号发起升级/复审

### 4.2 07 不负责的内容

07 不负责：

- `MarketAssumptions` 公式构造
- `GoalSolverParams / EVParams / RuntimeOptimizerParams` 校准公式
- 候选战略配置生成逻辑本身
- Monte Carlo 求解
- 候选动作生成
- EV 五项打分
- 决策卡文案细节与布局结构
- 任何直接面向用户的策略判断公式

### 4.3 与其他层的关系

推荐明确写为：

> 03/05 负责给 07 输入；  
> 07 负责把下游串起来；  
> 07 不反向侵入 02/04/09/10 的核心业务语义。

## 5. 07 的核心问题是什么

07 要解决的，不是“再写一层业务逻辑”，而是四个控制问题。

### 5.1 这次到底是什么场景

是首次建档、月度巡检、事件型处置还是季度复审。

### 5.2 这一轮需要哪些模块

不是每轮都要跑完整链路。

- 首次建档：03 → 05 → 08 → 02 → 09
- 月度巡检：03 → 05 → 04 → 09（通常复用既有 Goal Solver 基线）
- 事件触发：03 → 05 → 04 → 09
- 季度复审：03 → 05 → 08 → 02 → 04 → 09

### 5.3 这一轮能不能继续跑

这取决于：

- `SnapshotBundle.bundle_quality`
- `CalibrationResult.calibration_quality`
- 是否存在阻断级约束冲突
- 是否存在候选贫乏或强冷静期

### 5.4 这一轮结果如何落账和复盘

07 必须保证：

- 运行链条有唯一 `run_id`
- 各层结果都能回指上游版本
- 后续复盘能按 `run_id` 追到完整证据链

## 6. 07 的核心输入

07 的输入建议分成四组。

### 6.1 触发输入

包括：

- 定时器触发（每月 / 每季度）
- 用户手动触发
- 事件触发（结构偏离 / 行为 / 回撤 / 卫星超配）
- 管理员或人工复审触发

### 6.2 原始世界输入

包括：

- 03 所需五域原始输入
- `remaining_horizon_months`
- 当前账户上下文
- 行为日志 / 用户请求上下文

### 6.3 历史状态输入

包括：

- 上一次 `GoalSolverOutput`
- 上一次 `GoalSolverInput`
- 上一次 `CalibrationResult`
- 最近一次执行记录
- 最近一次人工 override 或冷静期记录

### 6.4 运行控制输入

包括：

- 是否允许本轮降级继续
- 是否允许人工覆盖既有基线
- 是否需要强制写日志 / 发告警
- 是否启用审计模式

## 7. 07 的核心输出

07 的正式输出不应是一堆散落对象，而应是一份统一的编排结果。

### 7.1 建议的输出对象

```python
from dataclasses import dataclass
from typing import Literal

class WorkflowType(Enum):
    ONBOARDING = "onboarding"
    MONTHLY = "monthly"
    EVENT = "event"
    QUARTERLY = "quarterly"

class WorkflowStatus(Enum):
    COMPLETED = "completed"
    BLOCKED = "blocked"
    DEGRADED = "degraded"
    ESCALATED = "escalated"

@dataclass
class OrchestratorOutput:
    run_id: str
    workflow_type: WorkflowType
    status: WorkflowStatus

    bundle_id: str | None
    calibration_id: str | None
    solver_snapshot_id: str | None

    snapshot_bundle: SnapshotBundle | None
    calibration_result: CalibrationResult | None
    goal_solver_output: GoalSolverOutput | None
    runtime_result: RuntimeOptimizerResult | None

    card_build_input: "DecisionCardBuildInput | None"

    blocking_reasons: list[str]
    degraded_notes: list[str]
    escalation_reasons: list[str]
```

### 7.2 输出对象的作用

它不是给用户直接看的。  
它是给以下两类对象消费的：

- 09：生成决策卡
- 基础设施层：持久化与审计

## 8. 07 的 workflow 拆分

v1 建议显式拆成四条 workflow，不要把所有情况塞进一个超级入口。

### 8.1 首次建档流（ONBOARDING）

输入：

- 全量五域原始输入
- 新账户画像与目标
- 无历史基线或历史基线作废

执行：

1. 03 构建 `SnapshotBundle`
2. 05 生成 `CalibrationResult`
3. 若 `bundle_quality / calibration_quality` 可接受，则调用 08 生成 `candidate_allocations`
4. 调用 02 完整求解 `GoalSolverOutput`
5. 组装 `DecisionCardBuildInput(card_type="goal_baseline")`
6. 调用 09 生成基线配置解释卡
7. 写入本轮基线与参数版本

输出重点：

- 初始目标基线
- 推荐战略配置
- 配置解释卡
- 运行基线的持久化版本锚点

### 8.2 月度巡检流（MONTHLY）

输入：

- 本轮快照与校准结果
- 上一次有效的 Goal Solver 基线
- 当前 live portfolio / 行为状态

执行：

1. 03 构建 `SnapshotBundle`
2. 05 生成 `CalibrationResult`
3. 检查质量与阻断条件
4. 复用最新有效 `GoalSolverOutput + GoalSolverInput`
5. 调用 04 执行 `RuntimeOptimizerMode.MONTHLY`
6. 组装 `DecisionCardBuildInput(card_type="runtime_action")`
7. 调用 09 生成月度决策卡
8. 写入本轮运行记录

输出重点：

- 本月是否需要动作
- 若需要，推荐动作与备选动作
- 若不需要，是否输出 `FREEZE / OBSERVE`

### 8.3 事件触发流（EVENT）

输入：

- 本轮快照与校准结果
- 事件子触发标志
- 用户请求上下文（如追热点请求、恐慌减仓请求）
- 当前有效基线

执行：

1. 03 构建 `SnapshotBundle`
2. 05 生成 `CalibrationResult`
3. 判断事件子触发：
   - structural_event
   - behavior_event
   - drawdown_event
   - satellite_event
4. 调用 04 执行 `RuntimeOptimizerMode.EVENT`
5. 必要时根据 `candidate_poverty / cooldown / behavior` 升级为人工复审
6. 组装 `DecisionCardBuildInput(card_type="runtime_action" or "blocked")`
7. 调用 09 生成事件型决策卡

输出重点：

- 当前事件为什么触发
- 系统本轮允许什么、不允许什么
- 推荐动作与禁止原因

### 8.4 季度复审流（QUARTERLY）

输入：

- 本轮快照与校准结果
- 最新画像与现金流信息
- 是否存在重大参数更新

执行：

1. 03 构建 `SnapshotBundle`
2. 05 生成 `CalibrationResult`
3. 调用 08 生成或刷新 `candidate_allocations`
4. 调用 02 完整重算 `GoalSolverOutput`
5. 调用 04 执行 `RuntimeOptimizerMode.QUARTERLY`
6. 组装 `DecisionCardBuildInput(card_type="quarterly_review")`
7. 调用 09 输出季度复审卡
8. 写入新的季度基线与参数版本

输出重点：

- 目标可达性是否变化
- 新基线与旧基线差异
- 本季度是否需要正式结构调整

## 9. 07 的阻断、降级与升级思路

07 必须是系统里唯一正式做“运行控制裁决”的层。

### 9.1 阻断来源

07 需要至少处理以下阻断来源：

#### A. 输入层阻断
来自 03：

- `bundle_quality == DEGRADED`
- 核心域缺失
- 目标期限非法
- 约束边界冲突

#### B. 校准层阻断
来自 05：

- `calibration_quality == "degraded"`
- Goal Solver 核心参数无法构造且无先验
- 约束解释存在不可继续的冲突

#### C. 运行期升级信号
来自 04：

- `candidate_poverty == True`
- 强冷静期 + 用户发起高风险动作请求
- QUARTERLY 模式下动作集贫乏或基线不稳定

### 9.2 阻断原则

推荐明确：

- **输入坏了，不要硬跑**
- **能降级时先降级，但必须显式打标**
- **不能降级时宁可输出 blocked card，也不要伪造动作建议**
- **升级优先于静默失败**

### 9.3 建议的状态转换

- `bundle_quality=DEGRADED` → `WorkflowStatus.BLOCKED`
- `calibration_quality=degraded` → `WorkflowStatus.BLOCKED`
- `calibration_quality=partial` → `WorkflowStatus.DEGRADED`
- `candidate_poverty=True` → `WorkflowStatus.DEGRADED or ESCALATED`
- 行为强冷静期命中 → 不一定阻断 workflow，但应强制卡片进入受限动作口径

## 10. 07 的版本绑定与可追踪性

07 的一个核心职责，是把这一轮运行的所有版本锚点绑在一起。

### 10.1 最低要求的版本链

建议至少保留以下链条：

- `run_id`
- `bundle_id`
- `calibration_id`
- `GoalSolverInput.snapshot_id`
- `GoalSolverParams.version`
- `RuntimeOptimizerParams.version`
- `EVParams.version`
- `RuntimeOptimizerResult.run_timestamp`

### 10.2 关键约束

推荐明确：

- 一个 `run_id` 只能引用一份 `bundle_id`
- 一个季度复审 run 只能引用本轮新生成的 solver 基线，不允许混入旧季度基线
- `GoalSolverInput.snapshot_id` 必须派生自本轮 `bundle_id`
- 09 输出的卡片也必须带 `run_id / bundle_id / calibration_id`，便于回放

## 11. 07 的实现思路

### 11.1 整体方向

07 的 v1 不应实现成“超级业务中枢”，而应实现成一个：

- 流程明确
- 责任收口
- 阻断透明
- 版本可追踪
- 对下游零侵入

的编排层。

07 的首要目标不是“自动化越多越好”，而是：

> **先让系统每一轮运行都按正确顺序、正确基线、正确参数去跑。**

### 11.2 三段式实现逻辑

#### 第一段：识别
识别触发源、判断 workflow、决定本轮需要哪些模块。

#### 第二段：准备
调用 03 / 05，拿到 `SnapshotBundle` 与 `CalibrationResult`，完成阻断/降级判断，并准备基线输入。

#### 第三段：执行与发布
调用 08 / 02 / 04 得到结构化结果，生成给 09 的 card build input，并统一落账。

## 12. 07 的建议接口

```python
def run_orchestrator(
    trigger: "TriggerSignal",
    raw_inputs: "RawInputEnvelope",
    prior_solver_output: GoalSolverOutput | None,
    prior_solver_input: GoalSolverInput | None,
    prior_calibration: CalibrationResult | None,
) -> OrchestratorOutput:
    """
    07 的唯一正式入口。

    输入：
    - 触发信号
    - 本轮原始输入包
    - 历史基线与历史校准结果（如存在）

    输出：
    - 本轮统一编排结果（可供 09 和持久化层消费）
    """
    ...
```

### 12.1 伪代码主流程

```python
def run_orchestrator(...) -> OrchestratorOutput:
    workflow_type = _select_workflow(trigger, prior_solver_output)

    bundle = build_snapshot_bundle(...)
    calibration = run_calibration(bundle, prior_calibration=prior_calibration, ...)

    control = _decide_control_flow(bundle, calibration, trigger)
    if control.blocked:
        return _build_blocked_output(bundle, calibration, control)

    if workflow_type == WorkflowType.ONBOARDING:
        allocs = generate_candidate_allocations(...)
        solver_output = run_goal_solver(...)
        card_input = _make_goal_baseline_card_input(...)
        return _finalize_output(...)

    if workflow_type == WorkflowType.MONTHLY:
        runtime_result = run_runtime_optimizer(... mode=MONTHLY ...)
        card_input = _make_runtime_card_input(...)
        return _finalize_output(...)

    if workflow_type == WorkflowType.EVENT:
        runtime_result = run_runtime_optimizer(... mode=EVENT ...)
        card_input = _make_runtime_card_input(...)
        return _finalize_output(...)

    if workflow_type == WorkflowType.QUARTERLY:
        allocs = generate_candidate_allocations(...)
        solver_output = run_goal_solver(...)
        runtime_result = run_runtime_optimizer(... mode=QUARTERLY ...)
        card_input = _make_quarterly_review_card_input(...)
        return _finalize_output(...)
```

## 13. 07 的持久化与审计思路

07 本身不定义数据库结构，但应该明确哪些东西必须被持久化。

### 13.1 必须落账的对象

- `run_id`
- `workflow_type`
- `bundle_id`
- `calibration_id`
- `GoalSolverInput.snapshot_id`
- `GoalSolverOutput`（若本轮有）
- `RuntimeOptimizerResult`（若本轮有）
- `DecisionCard` 或其序列化结果
- `blocking_reasons / degraded_notes / escalation_reasons`
- 用户是否执行、是否 override、override 原因

### 13.2 持久化原则

- 07 负责“写什么”
- 基础设施层负责“写到哪里、如何事务化、如何索引”
- 07 不应内嵌数据库驱动或网络 IO 细节到业务规格里

## 14. 07 与其他文档的关系

### 与 `03_snapshot_and_ingestion.md`
07 是 03 的直接控制消费者。  
03 只构建 bundle；07 决定是否继续跑下一步。

### 与 `05_constraint_and_calibration.md`
07 接收 `CalibrationResult` 并注入下游。  
05 只解释与校准，不决定 workflow。

### 与 `02_goal_solver.md`
07 决定何时调用 Goal Solver，何时复用旧基线。  
02 不决定自己是否该被调用。

### 与 `04_runtime_optimizer.md`
07 是 04 的唯一合法上游调用方。  
04 只接受已准备好的状态与基线，不自己调度 workflow。

### 与 `09_decision_card_spec.md`
07 不写文案，但必须明确给 09 什么类型的卡、带什么结构化输入。  
09 不负责决定 workflow，只负责把结果展示出来。

### 与 `10_ev_engine.md`
07 不应直接碰 10。  
10 只能通过 04 的正式运行期入口被间接触发。

## 15. 07 v1 范围

### 15.1 v1 应做到的

- 明确四条正式 workflow
- 明确 03 → 05 → 08/02/04 → 09 的串联关系
- 明确阻断、降级、升级的正式归属在 07
- 明确版本锚点绑定关系
- 明确对 09 的 card build input 交付职责
- 明确持久化与审计落账清单

### 15.2 v1 不应急着做的

- 复杂多账户并行编排
- 分布式任务调度平台
- 自动重试与故障转移中台
- 策略级自学习工作流生成
- 把 07 写成“万能系统中枢”
- 把 09 的展示逻辑和 07 混写

## 16. 07 的验收标准

07 的验收重点不应是“能调多少模块”，而应是：

### 16.1 路由正确性
能否在正确场景下走正确 workflow。

### 16.2 阻断透明性
出现坏输入、坏校准、坏候选时，是否显式阻断/降级，而不是静默继续。

### 16.3 版本一致性
一轮运行里的 bundle / calibration / solver / runtime / card 是否能被同一 `run_id` 串起来。

### 16.4 模块隔离性
07 是否没有越界去写求解、评分、文案与公式。

### 16.5 可复盘性
复盘时能否明确知道：

- 这轮为什么触发
- 走了哪条 workflow
- 用了哪版输入与参数
- 为什么推荐这个动作或为什么被阻断

---

## 17. 一句话收束

> **07 的价值，不是“把模块串一下”，而是把系统每一轮运行都收束成一条有触发、有控制、有版本、有证据、可阻断、可复盘的正式执行链。**
