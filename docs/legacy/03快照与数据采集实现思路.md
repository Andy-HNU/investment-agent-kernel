# 03_snapshot_and_ingestion.md

# 输入快照与采集层设计思路 v1

> **文档定位**：本文件描述系统的输入快照与采集层（Snapshot & Ingestion Layer）的作用、边界、输入输出、实现思路与 v1 范围。  
> 它是 `05_constraint_and_calibration.md` 的上游输入层，是 `07_orchestrator_workflows.md` 在 workflow 触发前的数据准备层。  
> 它不参与投资求解、不参与动作评分、不参与参数校准公式，只负责把原始输入世界整理成系统可消费的统一快照。

---

## 0. 一句话定义

**Snapshot & Ingestion Layer 是系统的输入底座。**

它负责从外部数据源、账户侧、用户输入、规则配置和行为日志中获取原始数据，完成统一化、快照化和版本化，向下游提供可复现的输入基线。

它不回答：

- 这个目标能不能达成
- 当前应该生成哪些候选动作
- 哪个动作更优
- 哪些参数该上调或下调

它只回答一件事：

> **当前这一轮系统运行，到底基于哪一份输入世界在工作。**

---

## 1. 为什么需要单独做 03

系统后续模块都在隐含依赖一个事实：

- `02_goal_solver.md` 需要明确版本的目标、现金流、当前资产值、市场假设
- `04_runtime_optimizer.md` 需要明确版本的账户状态、市场状态、约束状态、行为状态
- `10_ev_engine.md` 只接受 Runtime 给它的结构化状态和候选动作，不自己拉数据
- `05_constraint_and_calibration.md` 需要一个稳定、统一、可追踪的上游输入集合来做解释与校准

如果没有 03，会出现三个问题：

### 1.1 复盘不成立
你无法明确回答某次 Goal Solver 或某次 Runtime/EV 的判断到底是基于哪一组输入做出的。

### 1.2 模块边界被破坏
Goal Solver、Runtime、EV 会被迫偷偷长出“拿点数据、补点字段、修点格式”的职责。

### 1.3 校准层上游不稳
05 想做参数回写，但如果输入来源不统一、不定版，校准结果会漂。

因此 03 的职责不是“随手拉数据”，而是：

> **把外部世界里分散、异步、脏乱的数据，整理成系统内部统一、冻结、可追踪的输入基线。**

---

## 2. 03 在系统中的作用

03 的作用可概括为三层：

### 2.1 输入统一
把不同来源、不同时间、不同口径的数据统一成系统内部可识别的对象。

### 2.2 快照冻结
给每次系统运行准备一个输入截面，确保同一轮运行引用的是同一份输入基线。

### 2.3 版本锚点
让 05 的校准、02 的求解、04/10 的运行期评估都能回指到同一个输入版本。

所以 03 不是数据中台，不是指标工厂，而是：

> **输入世界 → 系统内部状态世界 的第一道门。**

---

## 3. 03 的系统位置

推荐系统链路如下：

外部输入  
→ `03_snapshot_and_ingestion`  
→ `05_constraint_and_calibration`  
→ `07_orchestrator_workflows`  
→ `02_goal_solver / 04_runtime_optimizer / 10_ev_engine`  
→ `09_decision_card_spec`

其中：

- 03 负责“拿输入、定版输入”
- 05 负责“解释输入、校准参数”
- 07 负责“选择时机、注入下游”
- 02/04/10 负责“消费结果”

---

## 4. 03 的职责边界

### 4.1 03 负责的内容

03 负责：

- 从外部来源接收原始输入
- 对原始输入做标准化与最小合法性校验
- 按域形成不可变快照
- 将多个域的快照组装成统一的输入 bundle
- 为单域快照和 bundle 分配可追踪标识
- 标记数据质量、完整性、时效性
- 将结果提供给 05 和 07 读取

### 4.2 03 不负责的内容

03 不负责：

- 计算 `MarketAssumptions`
- 识别 market regime
- 生成 `MarketState`
- 校准 EVParams / GoalSolverParams / RuntimeParams
- 生成候选动作
- 计算 GoalImpact
- EV 排序与打分
- 决策卡文案生成
- workflow 触发判断

### 4.3 与 05 的关系

03 和 05 是上下游关系，不是父子关系。

- **03** 提供原始快照与 bundle
- **05** 基于这些原始输入，构造 `MarketState`、`MarketAssumptions`、校准后的约束与参数

03 不应偷做 05 的事情；05 也不应直接越过 03 去对接原始外部源。

---

## 5. 03 到底采什么

建议按五个输入域组织。

### 5.1 市场域
市场域原始输入包括但不限于：

- 资产桶价格 / 净值时间序列
- 收益率序列
- 波动与相关性估计所需原始序列
- 估值原始输入
- 流动性原始输入
- 宏观 / 政策 / 利率 / 汇率等可选标签

注意：

- 03 只负责收集和整理这些原始输入
- 不在 03 中判断当前是否 risk-on / risk-off
- 不在 03 中直接生成 `MarketAssumptions` 或 `MarketState`

### 5.2 账户域
账户域输入包括：

- 当前持仓
- 权重
- 总资产
- 可用现金
- 账户 as-of 时间
- 最近调仓时间
- 已实现 / 未实现盈亏摘要

### 5.3 目标与现金流域
这部分输入包括：

- 目标金额
- 目标期限
- 目标优先级
- 成功率阈值
- 现金流计划
- 非规律现金流事件

### 5.4 约束域
这部分输入包括：

- IPS 硬约束
- 软偏好
- 资产桶分类映射
- 卫星上限
- 主题预算
- 再平衡带宽
- 禁止动作
- 冷静期约束

### 5.5 行为域
这部分输入包括：

- 历史操作日志
- 频繁交易统计
- 追涨 / 恐慌痕迹
- 冷静期记录
- 人工 override 历史
- 干预记录

---

## 6. 03 的核心对象

03 的核心对象建议只有两层：

### 6.1 单域快照
每个输入域各有一份快照：

- 市场快照
- 账户快照
- 目标快照
- 约束快照
- 行为快照

每个快照都应具备共同属性：

- as-of time
- source info
- ingest time
- schema version
- normalization version
- quality flags
- completeness 标记
- snapshot_id

### 6.2 输入 bundle
系统运行时真正引用的不是散落的多份输入，而是一份统一的输入 bundle。

bundle 的作用是说明：

> **这一轮系统运行，引用的是哪几份单域快照的组合。**

也就是说，bundle 是系统输入的一次“截面引用”。

## 7. 03 的实现思路

### 7.1 整体方向

03 的 v1 不应实现成一个庞大的数据平台，而应实现成一个轻量输入整理层：

- 先接有限几类输入源
- 先做最小标准化
- 先保证快照可追踪与可复盘
- 不急着做复杂的多源治理与特征工程

03 的 v1 目标不是“接入所有数据源”，而是：

> **先把系统真正依赖的输入世界定版。**

### 7.2 三段式实现逻辑

#### 第一段：采集
从外部源取得原始输入。  
来源可以是 API、文件、账户导出、手工输入、配置文件、行为日志。

#### 第二段：标准化
对原始输入做统一处理：

- 统一字段名
- 统一单位
- 统一时间格式
- 统一 bucket 名称
- 统一缺失值表达方式

注意：这一步是格式统一，不是业务解释。

#### 第三段：快照化
将标准化后的结果冻结成带标识的快照，并将一组可共同使用的快照组装成输入 bundle。

03 到这里应停止，不应继续向下解释成 `MarketAssumptions` 或 `MarketState`。

## 8. 03 的质量标记与降级思路

03 必须显式保留数据质量信息。

### 8.1 需要标记的问题
包括但不限于：

- 数据过旧
- 部分字段缺失
- 某域未准备完成
- 时间窗口不足
- 权重不完全闭合但可修复
- 行为域缺失
- 某些市场标签不可用

### 8.2 03 的原则
03 可以标记问题，但不负责最终业务决策。

也就是说：

- 03 负责说“这里不完整”
- 05 负责说“能不能继续解释”
- 07 负责说“这一轮 workflow 是否允许降级继续”

---

## 9. 03 v1 范围

### 9.1 v1 应做到的
- 明确五域输入对象
- 支持单域快照生成
- 支持统一 bundle 组装
- 支持 source metadata
- 支持 quality flags
- 支持 stale / partial / missing 的显式标记
- 支持下游按 bundle_id 回指输入基线

### 9.2 v1 不应急着做的
- 多主源自动仲裁
- 复杂数据修复
- 高级特征工程
- regime 识别
- 直接生成 `MarketAssumptions`
- 实时事件流平台
- 大而全数据中台
- 分布式缓存系统

---

## 10. 03 的验收标准

03 的验收重点不应是：

- 接了多少 API
- 字段有多少个
- 实时性吹得多高

真正的验收重点应是：

### 10.1 输入统一性
同一轮运行是否能明确引用同一份输入基线。

### 10.2 可回指性
02 / 04 / 10 / 05 是否都能回指到相同 bundle。

### 10.3 降级透明性
数据不完整时是否显式打标，而不是静默糊过去。

### 10.4 可复盘性
复盘时是否能知道当时系统到底看见了什么输入世界。

## 11. 与其他文档的关系

### 与 `02_goal_solver.md`
03 不直接给 02 原始市场数据；  
03 提供原始快照，05 再生成 Goal Solver 所需的 `market_assumptions`。  
02 需要能回指当前求解使用的是哪份 bundle。

### 与 `04_runtime_optimizer.md`
04 所需的 `MarketState / ConstraintState / BehaviorState / GoalSolverOutput / LivePortfolio`，其上游输入基线由 03 提供。  
04 本身不负责外部采集。

### 与 `05_constraint_and_calibration.md`
05 直接消费 03 的原始快照，并将其解释成校准后的参数与状态。  
05 是 03 的第一消费者，也是最重要的上游治理层。

### 与 `07_orchestrator_workflows.md`
07 需要基于某一轮 workflow 选择或引用合适的 bundle，再把 05 的产物注入 02/04/10。

### 与 `10_ev_engine.md`
EV 不直接读取原始外部输入。  
它只消费 Runtime 已构建好的 `EVState` 和候选动作。  
03 为这个链路提供输入底座，但不与 EV 直连。

---

## 12. 一句话收束

> **03 的价值，不是“把数据拉进来”，而是把系统每一次判断都锚定到一份明确、统一、可回放的输入世界。**
